## Failure analysis

### GLM-5.2 through ZCode desktop

`quoted-tokenizer` fixed unterminated quotes and trailing escape detection, but
still discarded an empty quoted token. The patch only added end-of-input
validation and did not track whether an empty token had begun.

`stable-toposort` correctly added duplicate-edge handling, cycle detection, and
deterministic heap ordering. It assigned all graph keys before dependency-only
nodes, however, so `build` was selected before the earlier dependency-only
node `sign`. One of seven held-out tests failed.

### GLM-5.2 through OpenCode Go

`quoted-tokenizer` treated backslashes as escapes only inside quotes and
preserved unknown escaped characters with their slash. The grader expected the
slash to be consumed for sequences such as `\t` and `\n`, and expected a
trailing pair of backslashes outside quotes to collapse to one. It also failed
to preserve an empty quoted token. Three of seven held-out tests failed.

`stable-toposort` made the same keys-first ordering choice as the desktop run.
The implementation otherwise handled duplicates, cycles, determinism, and
dependency-only nodes. One of seven held-out tests failed.

The `webdav-endpoint` agent process reached the 900-second limit, but its saved
patch passed every regression and held-out test. The result is scored as a task
pass with an agent timeout recorded separately.

### GPT-5.5 high through Codex CLI

`quoted-tokenizer` added explicit token-start tracking, so it preserved empty
quoted tokens. It still preserved the slash for unknown escapes and did not
collapse a trailing pair of backslashes outside quotes. Two of seven held-out
tests failed.

`stable-toposort` was concise and otherwise correct, but it also placed all
declared graph keys before dependency-only nodes. This produced `build` before
`sign` in the same held-out case. One of seven held-out tests failed.

## Code quality

GPT-5.5 high produced the most compact set of patches in this run: 307 changed
lines across all tasks, compared with 418 for GLM-5.2 through OpenCode Go and
466 for the manual GLM-5.2 run. On passing tasks alone, the totals were 237,
338, and 393 respectively.

The difference was clearest on the large Click task:

| Run | Files changed | Changed lines |
|---|---:|---:|
| GLM desktop | 2 | 206 |
| GLM Go | 3 | 125 |
| GPT-5.5 high | 2 | 47 |

GPT-5.5 high made the smallest focused change while preserving parameter
sources, default-map behavior, multi-hop forwarding, and immutable snapshots.
Its implementation introduced one internal dispatch helper and used
`MappingProxyType` directly.

The OpenCode Go patch was also structured and documented, and it updated the
changelog. It was more invasive because it added a private source-propagation
keyword to `Context.invoke`.

The desktop GLM patch was the most elaborate. It separated preparation and
default-source resolution into helpers and documented the behavior thoroughly,
but introduced substantially more internal machinery than the other passing
solutions.

Across the compact tasks, all three generally kept changes to one implementation
file. The failed topological-sort patches were readable and algorithmically
sound apart from the tie-break interpretation. The tokenizer patches show the
main quality gap: each addressed the obvious examples but did not derive a
complete escape-state specification.

## Interpretation and limitations

This run does not establish a correctness winner. The three routes had
identical task outcomes on an eleven-task suite.

GPT-5.5 high was faster than the automated GLM route in this environment and
produced smaller successful patches, but provider latency, model service load,
and agent implementation all affect wall-clock time.

The two shared failures should not be treated as uncontested model errors:

- The topological-sort prompt says to use input declaration order, but does not
  explicitly define how graph keys and dependency-only nodes share that order.
  All three chose keys first; the grader expects dependency-only nodes to
  retain first appearance while scanning dependency lists.
- The tokenizer prompt specifies escaped quotes and backslashes, but the
  grader also defines behavior for unknown escapes and trailing backslashes.

Future revisions should make those semantics explicit or classify the affected
tests as robustness probes rather than strict issue requirements.

## GPT-5.5 xhigh targeted rerun

The two tasks failed by GPT-5.5 high were rerun independently with xhigh
reasoning. This was not a full-suite run and is not included in the primary
pass-rate table.

| Task | High | Xhigh | High time | Xhigh time | High lines | Xhigh lines |
|---|---:|---:|---:|---:|---:|---:|
| quoted-tokenizer | Fail | Fail | 57 s | 86 s | 14 | 31 |
| stable-toposort | Fail | Fail | 69 s | 116 s | 56 | 56 |

Xhigh produced the same held-out outcomes:

- The tokenizer preserved empty quoted tokens but still retained the slash for
  unknown escapes and did not collapse a trailing pair of backslashes outside
  quotes. Two of seven held-out tests failed.
- The topological sorter again placed `build` before the dependency-only node
  `sign`. One of seven held-out tests failed.

The targeted xhigh run took 202 seconds, compared with 126 seconds for the same
two high-reasoning tasks, approximately 61% longer. It did not improve
correctness or the topological-sort patch size, and its tokenizer patch was
larger. On these two tasks, increasing reasoning effort did not help.
